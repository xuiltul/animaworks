# descriptionの書き方ガイド

descriptionはスキルの発動を決定する最重要フィールドである。
ユーザーのメッセージと description の内容がマッチした場合にのみ、スキル名が候補として返却され、必要時に本文が読み込まれる。
マッチングは**3段階（Tier 1 → Tier 2 → Tier 3）**の順で行われ、前のティアでマッチしなかったスキルのみが次のティアの対象になる。

## 3段階マッチングの概要

| ティア | 方式 | 条件 |
|--------|------|------|
| **Tier 1** | 括弧キーワード / 区切りキーワード | 「」内の語句、または `、` `。` `.` 改行で区切った語句（2〜20文字）がメッセージに**部分文字列**として含まれる |
| **Tier 2** | 語彙マッチ | descriptionから抽出した単語（3文字以上、ストップワード除外）のうち、**2語以上**がメッセージに出現 |
| **Tier 3** | ベクトル検索 | RAG retriever によるセマンティック類似度（類似度 0.88 以上、最大3件）※retriever が利用可能な場合のみ |

**重要**: 「」で囲んだキーワードがある場合、Tier 1 では括弧キーワードのみが使われ、カンマ区切りは無視される。括弧キーワードを明示すると、意図した語句だけをトリガーにできる。

## 最重要ルール: ドメイン固有に具体的に書く

汎用的・抽象的な記述は他のスキルと誤マッチを起こす。
必ずそのスキル固有の具体的な用語・操作・対象を明記すること。

**悪い例（汎用的 → 偽陽性を起こす）**:

```yaml
description: >-
  AnimaWorksサーバーの管理を行う。プログラム更新後のリロードやシステム状態の確認を担当する。
```

→ 「管理」「確認」「状態」が汎用的すぎて、無関係なメッセージにもマッチしてしまう。

**良い例（ドメイン固有 → 正確にマッチ）**:

```yaml
description: >-
  AnimaWorksサーバープロセスの運用管理スキル。
  コード更新後のホットリロード(server reload)、Animaプロセスの再起動、
  サーバーステータス確認(起動中Anima一覧・メモリ使用量)を実行する。
  「リロードして」「更新を反映して」「再読み込みして」「システム状態」「サーバー再起動」「プロセス確認」
```

→ 「ホットリロード」「server reload」「Animaプロセス」「メモリ使用量」が固有の語彙。「」内のキーワードが Tier 1 で優先マッチする。

## 基本構成

1. **1文目**: スキルの目的を**具体的な対象と操作**で述べる（「〜の管理」ではなく「〜のホットリロード、再起動、ステータス確認」）
2. **2文目以降**: 使用するツール名、API名、具体的な操作手順を記載する
3. **末尾**: `「」` 区切りのキーワードを列挙する（Tier 1 の主トリガー）

## キーワード設計のコツ

### 「」括弧キーワード（Tier 1 優先）

- ユーザーが自然に発話する短いフレーズ（2〜20文字）を選ぶ
- 同義語・類義語を含める（例: 「定期実行」「cron設定」「スケジュール」）
- 括弧キーワードがある場合、カンマ区切りは Tier 1 では使われないため、**意図した語句だけを括弧で囲む**と誤発動を抑えられる
- キーワードが多すぎると誤発動するため、3〜6個程度に絞る
- キーワードもドメイン固有にする（「確認」ではなく「cron確認」「Slack確認」）

### 括弧を使わない場合（Tier 1 フォールバック / Tier 2）

- `、` `。` `.` 改行で区切ると、2〜20文字の語句が Tier 1 のキーワードとして抽出される
- 英語などで括弧を使わない場合、Tier 2 に頼る。description に**2語以上の固有語**を含めるとマッチしやすい
- 英語は単語境界マッチのため、`document creation` のように複数の意味のある単語を入れると効果的

### 正規化

- メッセージと description は NFKC 正規化 + 小文字化されて比較される
- 全角 `ｃｒｏｎ` は半角 `cron` と同一としてマッチする

## description具体化チェック

以下の語彙が単独で使われていたら、より具体的な表現に置き換える:

| 避けるべき語彙 | 改善例 |
|--------------|--------|
| 管理を行う | ホットリロード、プロセス再起動、ステータス確認 |
| 確認する | slack_searchツールでDM取得、未読チェック |
| 対応を行う | chatwork_searchでルーム内未読取得、返信送信 |
| 仕組みと使い方 | RAG/Priming/Consolidation、実行モード(S/C/D/G/A/B) |
| 作成するためのメタスキル | YAMLフロントマター形式で作成、「」キーワード設計 |

## 例

```yaml
description: >-
  AnimaWorksのcronジョブ(APScheduler)の設定と管理を行うスキル。
  crontab形式での定期実行タスクの追加・編集・削除、
  次回実行時刻の確認、実行ログの参照手順を提供する。
  「cron設定」「定期実行」「スケジュール」「定時タスク」「cron確認」
```
