# Risk Auditor — machine 活用ガイド

Risk Auditor が machine を使って P&L 計算・運用健全性スキャンを実施するためのガイド。

> **重要**: machine は「データ収集・計算」に使う。「リスク判断・APPROVE/REJECT の決定」は Auditor 自身が行う。machine の出力を無検証で Director に渡すことは禁止。

---

## 基本ルール

1. **計画書ファースト**: machine に渡す前に「何を検証させるか」を明確にする
2. **出力は draft**: machine の出力はそのまま最終検証結果にしない。Auditor が判断を加える
3. **ファイルパス**: `state/plans/` に計画書を保存し、machine への instruction として渡す
4. **レート制限遵守**: machine の同時実行制限を守る
5. **機密データ注意**: machine には API キー・パスワードを渡さない

## Phase 1: P&L 検証（machine 活用）

### 用途と instruction テンプレート

#### 1-A: P&L 独立計算

```
以下のトレードログから P&L を独立計算せよ:

ログファイル: {パス}
期間: {開始日〜終了日}
計算項目:
- 総 P&L（手数料込み）
- 勝率（勝ち/負け/引き分けの件数）
- プロフィットファクター
- 最大ドローダウン（額・率・期間）
- 最大連続負け数
- Sharpe 比率

比較対象: {Director/Engineer が報告した数値}
出力: 計算結果 + 報告値との差異リスト
```

#### 1-B: 期待値 vs 実績の乖離分析

```
以下のバックテスト結果とライブ実績を比較せよ:

バックテスト: {backtest-report.md のパス}
ライブ実績: {トレードログのパス}
比較項目:
- P&L の乖離率
- 勝率の乖離
- 最大ドローダウンの乖離
- 取引頻度の乖離
- スリッページ実績 vs 想定

出力: 乖離サマリー + 乖離が閾値 {pl_divergence_threshold} を超える項目のフラグ
```

## Phase 2: 運用健全性スキャン（machine 活用）

#### 2-A: プロセス・接続確認

```
以下のサービスの稼働状態を確認せよ:

確認対象:
- {bot名1}: プロセス稼働確認 + 最終ログタイムスタンプ
- {bot名2}: プロセス稼働確認 + 最終ログタイムスタンプ
- API接続: {取引所名} API のレスポンス確認

出力: 各サービスのステータス（稼働中/停止/応答なし）+ 最終活動時刻
```

#### 2-B: 資産残高照合

```
以下の資産残高を照合せよ:

照合対象:
- {取引所/ウォレット1}: 期待残高 {金額} vs 実残高
- {取引所/ウォレット2}: 期待残高 {金額} vs 実残高
- 合計ポートフォリオ: 期待合計 {金額} vs 実合計

許容差異: {balance_tolerance}

出力: 照合結果 + 許容差異を超える項目のフラグ
```

#### 2-C: 約定検証

```
以下の期間の注文と約定を照合せよ:

期間: 直近{期間}
対象: {bot名/戦略名}

検証項目:
- 発注された注文数 vs 約定した注文数
- 部分約定の有無と影響
- 約定拒否/タイムアウトの有無
- 想定スリッページ vs 実際のスリッページ

出力: 約定サマリー + 異常フラグ
```

## Phase 3: Carry-forward 検証（machine 活用）

#### 3-A: Tracker 整合性チェック

```
以下の追跡表の整合性を検証せよ:

performance-tracker: {パス}
ops-issue-tracker: {パス}

検証項目:
- 前回レビューで「未解消」だった項目が全件言及されているか（silent drop 検出）
- ステータスの遷移が論理的か（例: 「解消」→「未対応」への逆行がないか）
- 新規追加された項目に初検出日・深刻度が付いているか

出力: 整合性チェック結果 + silent drop 検出リスト
```
