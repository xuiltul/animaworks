# Trading Engineer — machine 活用ガイド

Trading Engineer が machine を使って bot 実装・バックテスト・インフラ構築を実施するためのガイド。

> **重要**: 計画書なしの machine 実行は禁止。必ず MD 形式の計画書を事前に作成し、instruction として渡すこと。

---

## 基本ルール

1. **計画書ファースト**: machine に渡す前に計画書（impl.plan.md 相当）を作成する
2. **出力は draft**: machine の出力はそのまま本番にしない。Engineer が検証する
3. **ファイルパス**: `state/plans/` に計画書を保存し、machine への instruction として渡す
4. **ワークスペース指定**: machine には必ずワークスペースディレクトリを指定する（`-d {path}`）
5. **レート制限遵守**: machine の同時実行制限を守る
6. **本番環境への直接変更禁止**: machine に本番環境を直接変更させない

---

## Phase 1: 実装計画（Engineer が書く）

strategy-plan.md を受け取り、技術的な実装詳細を計画書にまとめる。

### impl.plan.md テンプレート

```markdown
# 実装計画書: {戦略名/タスク名}

status: draft
author: {anima名}
date: {YYYY-MM-DD}
type: impl-plan
source: {strategy-plan.md のパス}

## 概要
{strategy-plan.md のゴールに対する技術的アプローチ — 1〜3文}

## 実装ステップ
1. {ステップ1: 具体的な実装内容}
2. {ステップ2: 具体的な実装内容}
3. ...

## テスト計画
- [ ] {テスト1: ユニットテスト対象}
- [ ] {テスト2: 統合テスト対象}
- [ ] {テスト3: dry-run 検証}

## リスクパラメータの実装
| パラメータ | strategy-plan.md の値 | 実装方法 |
|-----------|---------------------|---------|
| 最大ドローダウン | {値} | {どこに実装するか} |
| ポジション上限 | {値} | {どこに実装するか} |
| 損切りルール | {条件} | {どこに実装するか} |

## ロールバック計画
{問題発生時の復旧手順}
```

## Phase 2: bot 実装（machine 活用）

### 用途と instruction テンプレート

#### 2-A: 戦略ロジック実装

```
以下の実装計画に従い、戦略ロジックを実装せよ:

計画書: {impl.plan.md のパス}
ワークスペース: {リポジトリパス}

実装要件:
- {言語/フレームワーク}
- ランダムシード固定（再現可能性）
- ログ出力（エントリー/エグジット/損益を記録）
- エラーハンドリング（API障害時のリトライ、ポジション保護）

テスト:
- ユニットテストを作成すること
- 既存テストが全件パスすること
```

#### 2-B: バックテスト実行

```
以下の戦略のバックテストを実行せよ:

戦略: {コードパス}
データ: {データパス/取得方法}
期間: {開始日〜終了日}
パラメータ: {パラメータセット}

バックテスト条件:
- スリッページ: {想定値}
- 手数料: {取引所手数料率}
- 初期資金: {金額}

出力:
- P&L 推移
- 勝率・プロフィットファクター
- 最大ドローダウン（額・率・期間）
- 取引回数と平均保有期間
- Sharpe比率
```

## Phase 3: インフラ変更（machine 活用）

### 用途: デプロイ・設定変更

```
以下のインフラ変更を実施せよ:

計画書: {impl.plan.md のパス}
対象: {サーバー/サービス}

変更内容:
- {変更1}
- {変更2}

検証手順:
- {検証1: 変更後の動作確認方法}

ロールバック手順:
- {問題発生時の復旧方法}
```

> **注意**: 本番環境の変更は dry-run で検証済みであること。machine に本番の API キーを渡さない。
