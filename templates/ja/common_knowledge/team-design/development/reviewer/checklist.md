# Reviewer — 品質チェックリスト

machine 使用有無に関わらず、Reviewer がレビュー品質を担保するためのチェックリスト。

---

## レビュー観点の網羅性

### 要件充足

- [ ] plan.md の完了条件が全て満たされているか
- [ ] plan.md の制約条件に違反する箇所がないか
- [ ] スコープ外の変更が含まれていないか

### コード品質

- [ ] 命名規約がプロジェクトの慣習に従っているか
- [ ] 関数・メソッドの責務が単一か（長すぎる関数がないか）
- [ ] 重複コードがないか（DRY 原則）
- [ ] 不要なコード（デバッグ出力、コメントアウト、未使用 import）がないか

### 設計適合性

- [ ] 既存のアーキテクチャ・パターンと整合しているか
- [ ] 適切な抽象化レベルか（過度な抽象化・不足のいずれでもないか）
- [ ] 依存関係の方向が正しいか（循環依存がないか）
- [ ] 公開 API の変更がある場合、後方互換性が考慮されているか

### セキュリティ

- [ ] 外部入力のバリデーションが適切か
- [ ] SQL インジェクション・XSS 等の脆弱性がないか
- [ ] 秘密情報（API キー、パスワード等）がハードコードされていないか
- [ ] ファイル操作にパストラバーサルの危険がないか

### パフォーマンス

- [ ] N+1 クエリや不要なループがないか
- [ ] 大量データ処理時のメモリ使用量が考慮されているか
- [ ] 非同期処理が適切に使われているか

### テスト

- [ ] 新規コードに対するテストが追加されているか
- [ ] テストケースが正常系・異常系を網羅しているか
- [ ] テストが脆弱でないか（タイミング依存、外部依存等）

---

## メタレビューの品質

### machine レビュー結果の検証

- [ ] 誤検出（False Positive）を除去したか
- [ ] 見落とし（False Negative）を補完したか
- [ ] 重要度の判定（Critical/Warning/Info）が妥当か
- [ ] 指摘内容が事実に基づいているか（推測や思い込みでないか）

### フィードバックの品質

- [ ] Critical 指摘に具体的な修正案が含まれているか
- [ ] 指摘の理由が明記されているか（「なぜ問題なのか」）
- [ ] 建設的なトーンになっているか（問題の指摘 + 改善提案）
- [ ] 再現可能な形式で指摘されているか（ファイル名、行番号）
