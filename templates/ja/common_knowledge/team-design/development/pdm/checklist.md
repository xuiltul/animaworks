# PdM — 品質チェックリスト

machine 使用有無に関わらず、PdM が成果物の品質を担保するためのチェックリスト。

---

## 調査報告書（investigation.md）の品質

### 内容の完全性

- [ ] 調査目的に対する回答が網羅されているか
- [ ] 事実と推測が明確に区別されているか
- [ ] 影響範囲の評価が含まれているか
- [ ] 定量的なデータ（ファイル数、行数、影響ユーザー数等）が含まれているか

### 信頼性

- [ ] 情報のソース（コード、ログ、ドキュメント）が明記されているか
- [ ] 「未確認」「要追加調査」の箇所が明示されているか
- [ ] 古い情報や廃止済みの仕様に基づいた記述がないか

---

## 計画書（plan.md）の品質

### ゴールの明確性

- [ ] ゴールが1〜3文で明確に記述されているか
- [ ] 「何をもって完了とするか」が客観的に判定可能か
- [ ] スコープ（やること / やらないこと）が明確か

### 実装方針の妥当性

- [ ] 調査結果に基づいた根拠があるか
- [ ] 技術的に実現可能な方針か（不明なら Engineer に事前確認済みか）
- [ ] 既存コード・アーキテクチャとの整合性が取れているか
- [ ] 段階的な実装が可能な構成になっているか

### 制約条件

- [ ] コーディング規約への準拠要件が記載されているか
- [ ] 既存 API との後方互換性が考慮されているか
- [ ] パフォーマンス要件があれば明記されているか

### 完了条件

- [ ] 各条件が客観的に検証可能か（「良くする」ではなく「X を Y にする」）
- [ ] テスト方針が含まれているか
- [ ] Reviewer / Tester が検証できる形式になっているか

### リスク管理

- [ ] 主要なリスクが洗い出されているか
- [ ] 各リスクに対する対策が記載されているか
- [ ] ロールバック方針が考慮されているか

---

## 委譲時の品質

### delegate_task の前に

- [ ] plan.md の `status: approved` を確認したか
- [ ] 完了条件が Engineer にとって実行可能な形式か
- [ ] 不明点を残したまま委譲していないか
- [ ] 期限（deadline）を設定したか

### フィードバック対応

- [ ] Reviewer / Tester からのフィードバックを全件確認したか
- [ ] Critical 指摘に対する対応方針を決定したか
- [ ] 修正後の再検証を依頼したか
