# チーム設計ガイド — 基本原則

## 概要

このドキュメントは、Anima チームを目的別に設計するための基本原則を定める。
フレームワークの組織機構（`organization/roles.md`、`organization/hierarchy-rules.md`）とは別に、
**「どのような目的に対して、どのようなロール構成でチームを組むか」** を設計するためのガイドである。

---

## なぜ機能分離するのか

AI エージェントにおけるロール分離には、人間のチームとは異なる固有の理由がある:

| 理由 | 説明 |
|------|------|
| **コンテキスト汚染の防止** | 1つのエージェントに全工程を担わせると、コンテキストウィンドウが肥大化し判断精度が落ちる。ロール分離によりコンテキストを限定できる |
| **専門性の深化** | ロール固有の指針・チェックリスト・記憶を持たせることで、汎用エージェントより高い品質を実現できる |
| **並行実行** | 独立した観点のロール（例: レビューとテスト）は同時に実行でき、スループットが向上する |
| **品質の構造的担保** | 「実行する者」と「検証する者」を分けることで、セルフレビューの盲点を排除できる |

---

## 設計の基本原則

### 1. 単一責任（Single Responsibility）

1つのロールには1つの明確な責務を持たせる。
責務が曖昧なロールは判断基準が不明確になり、品質が低下する。

**良い例**: 「コードレビューの実施と品質判定」（Reviewer）
**悪い例**: 「実装とレビューとテスト」（全部入り）

### 2. 実行と検証の分離

machine（外部エージェント）が実行した出力を、同じ Anima が検証する構造にする。
machine 出力を無検証で次工程に流すことは禁止。

```
Anima が指示書を書く → machine が実行 → Anima が検証・承認
```

### 3. ドキュメント駆動のハンドオフ

ロール間の引き継ぎは、ステータス付きのドキュメント（Markdown）で行う。
口頭（メッセージ本文のみ）での引き継ぎは情報が欠落するため避ける。

```
plan.md (status: approved) → delegate_task で Engineer に渡す
```

### 4. 並行実行可能な設計

独立した観点のロールは並行実行できるよう設計する。
依存関係がないロール同士を直列に配置しない。

## チーム設計プロセス

### Step 1: 目的を定義する

チームが達成すべきゴールを1文で明確にする。

例:
- 「ソフトウェア開発プロジェクトを、計画から実装・検証まで一貫して遂行する」
- 「カスタマーサポートの一次対応を24時間体制で行う」

### Step 2: ロールを分解する

目的達成に必要な機能を洗い出し、単一責任の原則に基づいてロールに分解する。

判断基準:
- その機能は **判断を伴うか** → 独立ロールにする価値がある
- その機能は **他と独立して実行できるか** → 分離して並行実行の恩恵を得る
- その機能は **専門的な知識を要するか** → 専門ロールにして品質を上げる

### Step 3: 責任境界を明確にする

各ロールの MUST（必ずやること）・SHOULD（推奨）・MAY（任意）を定義する。
特に隣接ロール間の境界を明確にする（「ここまでは自分、ここからは相手」）。

### Step 4: ハンドオフチェーンを設計する

ロール間のドキュメント受け渡し順序と、並行実行ポイントを図示する。

### Step 5: ロールテンプレートを選定する

フレームワークの `--role` オプション（engineer, manager, writer, researcher, ops, general）から
最も近いものを選び、`injection.md` でチーム設計のロール定義を上書きする。

## 兼務の判断基準

小規模チームやリソース制約がある場合、1人の Anima が複数ロールを兼務することがある。

### 兼務してよいケース

- **タスクの規模が小さい**: 全工程を1人で回しても品質が維持できる
- **観点が近い**: PdM + Engineer（計画と実装が密結合する小改修）
- **コスト制約**: 専用 Anima を立てるほどの作業量がない

### 分離すべきケース

- **実行と検証が同一人物**: Engineer が自分のコードをレビューする状態は避ける
- **コンテキストが競合する**: レビュー観点と実装観点の切り替えが頻繁に必要
- **並行実行の恩恵が大きい**: レビューとテストを同時に走らせたい場合

### 兼務時の注意

兼務する場合も、**ロールの切り替えを意識的に行う**こと。
「今は Reviewer として判断している」「今は Engineer として実装している」を明示する。

---

## 規模のスケーリング

| 規模 | 構成 | 適用場面 |
|------|------|---------|
| **ソロ** | 1 Anima が全ロール兼務 | 小タスク、プロトタイプ |
| **ペア** | PdM + Engineer（レビューは Engineer が兼務） | 中規模の定型タスク |
| **フルチーム** | PdM + Engineer + Reviewer + Tester | 本格的なプロジェクト |
| **スケールド** | PdM + 複数 Engineer + 複数 Reviewer + Tester | 大規模・複数モジュール |

スケールアップの判断基準:
- タスクの失敗コストが高い → ロール分離を増やす
- 並行作業可能なモジュールが多い → Engineer を増やす
- 品質要求が高い → Reviewer と Tester を独立させる
