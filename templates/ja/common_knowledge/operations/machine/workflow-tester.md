# machine ワークフロー — Tester（テスター）

## ロール定義

Tester は **テスト戦略の策定と machine によるテスト実行、結果の検証判断** を担うロールである。

- テスト戦略・観点の設計 → Tester 自身が判断
- テストケースの具体化 → machine に委託し、Tester が検証
- テストの実行 → machine に委託
- テスト結果の解釈・判定 → Tester 自身が判断

> 前提: `operations/machine/tool-usage.md` のメタパターンと共通原則を理解していること。

## テストの種類

テスト種類はプロジェクトと対象システムに依存する。
Tester は plan.md と対象システムの特性から適切なテスト戦略を判断すること。

代表的なテスト種類:

| 種類 | 概要 | 例 |
|------|------|-----|
| **E2E テスト** | 実際のユーザーフローをブラウザ操作等で検証 | ログインフロー、購入フロー |
| **コンポーネントテスト** | 実行環境内で個別コンポーネントの動作を検証 | API エンドポイント、バッチ処理 |
| **統合テスト** | 複数コンポーネント間の連携を検証 | DB → API → フロントエンド |
| **回帰テスト** | 既存機能が変更で壊れていないことを検証 | 既存テストスイートの実行 |

## ワークフロー

### Step 1: テスト計画書を作成する（Tester 自身が書く）

plan.md と実装内容を読み、テスト戦略・観点・シナリオの概要を計画する。

```bash
write_memory_file(path="state/plans/{date}_{概要}.test-plan.md", content="...")
```

### Step 2: machine にテストケース具体化を投げる

```bash
animaworks-tool machine run \
  "$(cat $ANIMAWORKS_ANIMA_DIR/state/plans/{test-plan.md})" \
  -d /path/to/worktree
```

結果を `state/plans/{date}_{概要}.test-cases.md` として保存する（`status: draft`）。

### Step 3: テストケースを検証する

Tester が test-cases.md を読み、以下を確認する:

- [ ] plan.md の完了条件がテストケースでカバーされているか
- [ ] 正常系・異常系の両方が含まれているか
- [ ] エッジケース（境界値、空値、大量データ等）が考慮されているか
- [ ] テストケースが実際に実行可能か（環境・前提条件が現実的か）
- [ ] 優先度の高いシナリオが網羅されているか

問題があれば Tester 自身が修正し、`status: approved` に変更する。

### Step 4: machine にテスト実行を投げる

approved の test-cases.md をベースに、machine にテスト実行を依頼する。

```bash
animaworks-tool machine run \
  "以下のテストケースを実行し、結果を報告せよ。
  各テストケースについて Pass/Fail とエビデンス（出力・スクリーンショット情報等）を記録すること。

  $(cat $ANIMAWORKS_ANIMA_DIR/state/plans/{test-cases.md})" \
  -d /path/to/worktree
```

結果を `state/plans/{date}_{概要}.test-report.md` として保存する（`status: draft`）。

### Step 5: テスト結果を検証する

Tester が test.report.md を読み、以下を判断する:

- [ ] 全テストケースの結果が記録されているか
- [ ] Fail の原因が特定されているか（実装の問題 vs テスト環境の問題 vs テストケースの問題）
- [ ] 偽陽性（実際は問題ないのに Fail）がないか
- [ ] 偽陰性（実際は問題あるのに Pass）の可能性がないか
- [ ] テスト結果から追加テストが必要な領域がないか

Tester 自身が所見を追記し、`status: approved` に変更する。

### Step 6: フィードバック

- 全テスト Pass → Engineer / PdM に合格を報告
- Fail あり → Engineer に具体的な不具合情報（再現手順・期待値・実際値）を報告
- 追加テスト必要 → Step 1 に戻る

## 制約事項

- テスト計画書（テスト戦略・観点）は MUST: Tester 自身が書く
- test-cases.md は machine が生成するが、Tester が検証・承認するまでドラフト扱い
- テスト結果を検証せずに「合格」と判定してはならない（NEVER）
- Fail の原因が不明な場合は、追加調査してから判定する（MUST）
- `status: approved` の付いていない test.report.md をフィードバックしてはならない（NEVER）
