# 에스컬레이션 플로차트

문제가 발생했을 때 "스스로 해결해야 하는지" "누군가에게 상담/보고해야 하는지"를 판단하기 위한 플로차트입니다.

판단에 망설여질 때 이 문서를 참조하세요. 분명히 스스로 해결할 수 있는 문제에는 이 플로차트가 불필요합니다.

---

## 판단 플로차트

문제가 발생하면 다음 단계를 순서대로 진행합니다.

### Step 1: 문제의 유형을 특정합니다

문제를 다음 중 하나로 분류합니다:

| 유형 | 설명 | 예시 |
|------|------|------|
| **A. 기술적 문제** | 도구나 시스템 동작에 관한 문제 | 도구 에러, 권한 부족, 파일 부재 |
| **B. 업무적 문제** | 태스크 진행 방법이나 판단에 관한 문제 | 사양 불명, 우선순위 판단, 차단 |
| **C. 대인적 문제** | 다른 Anima와의 연계에 관한 문제 | 답장 없음, 지시 상충, 담당 불명 |
| **D. 긴급 문제** | 즉각 대응이 필요한 문제 | 데이터 손실 위험, 보안 우려 |

### Step 2: 긴급도를 판정합니다

| 긴급도 | 기준 | 대응 |
|--------|------|------|
| **높음** | 방치하면 데이터 손실/보안 리스크가 있음 | MUST: 즉시 상사에게 보고 |
| **높음** | 다른 Anima의 작업이 완전히 멈춤 | MUST: 즉시 상사에게 보고 |
| **중간** | 자신의 작업이 차단되지만 다른 태스크에 착수 가능 | SHOULD: 1시간 이내에 상사에게 보고 |
| **낮음** | 작업 효율이 떨어지지만 진행은 가능 | MAY: 다음 heartbeat에서 보고 |

**긴급도가 "높음"인 경우** → Step 5로 진행 (즉시 에스컬레이션)

### Step 3: 스스로 해결을 시도합니다

다음 절차로 자력 해결을 시도합니다. 각 단계에서 해결되면 그 시점에서 종료합니다.

1. **메모리를 검색합니다**
   ```
   search_memory(query="문제 관련 키워드", scope="all")
   ```
   - 과거에 같은 문제를 경험한 적이 없는지 확인합니다
   - 절차서(procedures/)에 대처 방법이 없는지 확인합니다
   - 직전에 실행한 도구 결과나 송수신 메시지를 떠올리려면 `search_memory(query="...", scope="activity_log")`를 사용합니다 (통합 활동 로그의 최근 행동을 키워드로 회상)

2. **공유 지식을 검색합니다**
   ```
   search_memory(query="문제 관련 키워드", scope="common_knowledge")
   ```
   - `troubleshooting/common-issues.md`에 해당하는 문제가 없는지 확인합니다

3. **다른 접근 방식을 검토합니다**
   - 목표를 달성할 대체 수단이 없는지 생각합니다
   - 권한 부족이면 다른 경로, 도구 에러면 다른 도구

4. **자력 해결의 제한 시간**
   - 기술적 문제: 15분 이내에 해결되지 않으면 에스컬레이션
   - 업무적 문제: 판단에 망설여지면 즉시 에스컬레이션 (잘못된 판단의 리스크를 회피)
   - 대인적 문제: 1회 재시도 후 에스컬레이션

### Step 4: 에스컬레이션 대상을 결정합니다

| 문제 유형 | 우선 상담 상대 | 상담으로 해결되지 않는 경우 |
|----------|--------------|--------------------------|
| **A. 기술적 문제** | 동료 (같은 전문 분야인 경우) | 상사 |
| **B. 업무적 문제** | 상사 | — |
| **C. 대인적 문제** | 상사 (중재 요청) | — |
| **D. 긴급 문제** | 상사 (즉시) | — |

**판단 기준:**
- 동료에게 상담해도 되는 조건: 같은 상사를 가진 동료이며, 상대의 전문 분야에 관련된 문제인 경우
- 상사에게 보고 MUST인 조건: 업무 판단 필요, 타 부서 관여, 긴급도 높음
- 타 부서 Anima에게는 직접 연락하지 않음 (MUST: 상사 경유)

### Step 5: 에스컬레이션을 실행합니다

보고 메시지에는 다음 요소를 MUST 포함합니다:

1. **상황**: 무엇이 발생하고 있는지
2. **원인**: 무엇이 원인이라고 생각되는지 (불명이면 "원인 조사 중")
3. **시도**: 스스로 무엇을 시도했는지
4. **요청**: 상사에게 무엇을 해 주길 원하는지 (판단, 권한 부여, 중재 등)

**send_message 제약 (구현 준수)**:
- `intent`는 MUST: `report`(보고), `question`(질문) 중 하나를 지정합니다. 생략 불가. 태스크 위임에는 `delegate_task`를 사용
- acknowledgment(확인 응답), 감사, FYI는 DM 불가. Board(post_channel)를 사용합니다
- 1 run당 DM 수신자 수는 역할/status.json으로 설정된 상한 (general/ops는 2명, engineer는 5명, manager는 10명 등). 동일 수신자에게 1통만. 상한을 초과하는 전달은 Board를 사용합니다
- DM과 Board는 **동일한 아웃바운드 예산**을 공유합니다 (시간당/24시간당 제한 있음). 상세 내용은 `communication/sending-limits.md` 참조
- **수신자**: Anima 이름 또는 사람 별칭 (config에서 설정된 경우 Slack/Chatwork 등으로 외부 배달)
- **채팅 중**: 사람 사용자에게의 답변은 직접 텍스트로 합니다. send_message는 다른 Anima에게만 사용합니다
- **사람에게 연락** (미설정 수신자): `call_human`을 사용합니다
- 스레드 답신 시 `reply_to`와 `thread_id`를 지정하여 문맥을 유지합니다
- 긴급도가 "높음"이며 사람의 즉시 대응이 필요한 경우 `call_human`을 검토합니다 (subject, body, priority)

**post_channel (Board) 제약** (3명 이상에게 전달 시 사용):
- 메타 미설정 channel(general, ops 등)은 전원 사용 가능. 멤버 제한 channel은 멤버만 게시 가능(ACL). 접근 권한이 없는 경우 `manage_channel(action="info", channel="channel_name")`로 멤버를 확인할 수 있습니다
- 동일 channel에 1 run당 1개 게시까지. 동일 channel에 연속 게시는 쿨다운(`config.json`의 `heartbeat.channel_post_cooldown_s`, 기본값 300초)이 필요합니다
- 본문에 `@이름`으로 멘션 가능. 멘션 대상에게는 DM 알림이 전달됩니다

**call_human 파라미터**:
- `subject`, `body`는 필수. `priority`는 선택 (`low` / `normal` / `high` / `urgent`, 기본값 `normal`)

## 에스컬레이션 메시지 템플릿

### 템플릿 1: 차단 보고

```
send_message(
    to="supervisor_name",
    content="""[차단 보고]

■ 상황
태스크 "월간 리포트 작성"이 차단되었습니다.

■ 원인
매출 데이터가 저장된 /data/sales/ 디렉터리에 대한 읽기 권한이 없습니다.

■ 시도한 것
- permissions.json 확인 → /data/sales/는 미허가
- 대체 데이터 소스 검색 → 해당 없음

■ 요청
/data/sales/에 대한 읽기 권한 추가를 부탁합니다.""",
    intent="report"
)
```

### 템플릿 2: 판단 요청

```
send_message(
    to="supervisor_name",
    content="""[판단 요청]

■ 상황
태스크 "고객 대응 플로우 개선"에서 2가지 방안이 있습니다.

■ 선택지
A안: 기존 플로우를 점진적으로 수정 (공수: 소, 리스크: 저, 효과: 중)
B안: 플로우를 전면 재설계 (공수: 대, 리스크: 중, 효과: 고)

■ 제 의견
A안을 추천합니다. 이유: 현행 플로우의 문제점은 제한적이며, 점진적 수정으로 충분히 대응 가능합니다.

■ 요청
방침 결정을 부탁합니다.""",
    intent="question"
)
```

### 템플릿 3: 동료에게 기술 상담

```
send_message(
    to="peer_name",
    content="""[기술 상담]

Slack API의 rate limit에 걸리고 있습니다.

■ 상황
- 100건 이상의 메시지를 일괄 전송하려고 함
- 50건째 부근에서 429 Too Many Requests가 반환됨

■ 질문
Slack API의 rate limit 회피 방법에 대한 경험이 있으신가요?
배치 처리 간격을 두는 방법을 검토 중이지만 적절한 간격을 모르겠습니다.""",
    intent="question"
)
```

### 템플릿 4: 긴급 보고

긴급도가 "높음"이며 사람의 즉시 대응이 필요한 경우 `call_human`도 병행합니다.

```
send_message(
    to="supervisor_name",
    content="""[긴급 보고]

■ 상황
외부 API(XXX 서비스)에서 인증 에러가 지속적으로 발생하고 있습니다.

■ 영향
- YYY 태스크가 완전히 정지
- ZZZ 태스크도 같은 API를 사용하며 영향 가능성 있음

■ 시도한 것
- 재시도 3회 실시 → 모두 실패
- API 키의 유효성은 스스로 확인 불가

■ 요청
API 키 확인과 영향 범위 조사를 부탁합니다.""",
    intent="report"
)
```

사람에게 즉시 알림이 필요한 경우:
```
call_human(
    subject="[긴급] 외부 API 인증 에러 지속 발생",
    body="XXX 서비스에서 인증 에러가 지속되고 있습니다. YYY 태스크 정지 중. API 키 확인을 부탁합니다.",
    priority="urgent"
)
```

## 자주 있는 에스컬레이션 시나리오

### 시나리오 1: 지시 내용이 불명확

**상황**: 상사로부터 "리포트를 만들어 줘"라고 지시받았지만, 대상 기간, 형식, 제출처가 불명확함.

**올바른 대응**:
1. 스스로 추정할 수 있는 범위를 정리합니다
2. 불명확한 점을 구체적으로 질문합니다

```
send_message(
    to="supervisor_name",
    content="""리포트 작성 건에 대해 다음을 확인하겠습니다.

1. 대상 기간: 이번 달인가요?
2. 형식: 전번과 같은 Markdown 형식으로 괜찮은가요?
3. 제출처: knowledge/에 저장하면 되나요?

위 내용으로 문제없으면 착수하겠습니다.""",
    intent="question"
)
```

**해서는 안 되는 대응**:
- 확인 없이 독자적 해석으로 작업을 진행
- "지시가 불명확합니다"라고만 답함 (구체적인 질문이 없음)
- `send_message`에서 `intent`를 생략 (`report` / `question` 중 하나가 필수. 생략하면 에러)

### 시나리오 2: 여러 상사로부터 상충되는 지시

**상황**: 직속 상사로부터 "A를 우선해"라고 지시받았지만, 다른 Anima로부터 "B를 먼저 해 줘"라고 요청받음.

**올바른 대응**:
1. 직속 상사의 지시를 우선합니다 (MUST)
2. 상황을 직속 상사에게 보고합니다

```
send_message(
    to="direct_supervisor_name",
    content="""[우선순위 확인]

현재 태스크 A에 착수 중인데, XXX 씨로부터 태스크 B의 우선 요청이 있었습니다.
지시대로 태스크 A를 우선하여 진행하겠습니다만, 문제없나요?

태스크 B 요청 내용: YYY 대응 (XXX 씨의 요청)""",
    intent="question"
)
```

### 시나리오 3: 작업 중 에러가 연속 발생

**상황**: Chatwork API 메시지 전송이 3회 연속 실패함.

**올바른 대응**:
1. 에러 내용을 기록합니다
2. 3회 재시도해도 해결되지 않으면 에스컬레이션합니다
3. 차단되지 않은 다른 태스크에 착수합니다

```
# 에러 기록
write_memory_file(
    path="state/current_state.md",
    content="## 차단 중\n\nChatwork API 연속 에러\n- 1회째: 10:00 - 500 Internal Server Error\n- 2회째: 10:05 - 500 Internal Server Error\n- 3회째: 10:10 - 500 Internal Server Error\n\n상사에게 보고 완료. 다른 태스크에 착수 중.",
    mode="overwrite"
)

# 상사에게 보고
send_message(
    to="supervisor_name",
    content="[차단 보고] Chatwork API가 3회 연속 500 에러를 반환하고 있습니다. 외부 장애 가능성이 있습니다. 복구를 기다리면서 다른 태스크에 착수합니다.",
    intent="report"
)
```

### 시나리오 4: 자신의 책임 범위 밖의 문제를 발견

**상황**: 작업 중에 다른 Anima가 관리하는 데이터에서 불일치를 발견함.

**올바른 대응**:
1. 발견 사실을 기록합니다
2. 직속 상사에게 보고합니다 (타 부서 Anima에게 직접 연락하지 않음)

```
send_message(
    to="supervisor_name",
    content="""[정보 공유]

작업 중 다음과 같은 불일치를 발견했습니다. 제 담당 외이지만 공유합니다.

■ 발견 내용
/data/reports/monthly.md의 매출 합계와 /data/sales/summary.md의 값이 일치하지 않습니다.
- monthly.md: ¥1,234,567
- summary.md: ¥1,234,000

■ 발견 경위
월간 리포트 작성 중 데이터를 참조하면서 알게 되었습니다.

대응 여부는 맡기겠습니다.""",
    intent="report"
)
```

## 판단에 망설여질 때의 체크리스트

다음 중 하나라도 해당되면 MUST로 에스컬레이션하세요:

- [ ] 이 판단을 잘못하면 돌이킬 수 없는 영향이 있음
- [ ] 자신의 권한 범위를 넘는 조작이 필요함
- [ ] 타 부서의 Anima에 영향이 미침
- [ ] 15분 이상 해결책을 찾지 못함
- [ ] 같은 문제가 2회 이상 발생함
- [ ] 보안이나 데이터 안전성에 관련됨

다음에 해당하면 자력 해결을 MAY로 시도해도 됩니다:

- [ ] 과거에 유사한 문제를 해결한 경험이 있음
- [ ] 절차서(procedures/)에 대처 방법이 기재되어 있음
- [ ] 공유 지식(common_knowledge/)에 해결 방법이 있음
- [ ] 자신의 권한 범위 내에서 완결됨
- [ ] 실패해도 영향이 제한적임
