# Risk Auditor — machine 활용 가이드

Risk Auditor가 machine으로 P&L 계산·운용 건전성 스캔을 수행하기 위한 가이드.

> **중요**: machine은 「데이터 수집·계산」에 쓴다. 「리스크 판단·APPROVE/REJECT 결정」은 Auditor 본인이 한다. machine 출력을 무검증으로 Director에 넘기는 것은 금지.

---

## 기본 규칙

1. **계획서 우선**: machine에 넘기기 전에 「무엇을 검증할지」를 명확히 한다
2. **출력은 draft**: machine 출력을 그대로 최종 검증 결과로 하지 않는다. Auditor가 판단을 더한다
3. **파일 경로**: `state/plans/`에 계획서를 저장하고 machine에 넘기는 instruction으로 쓴다
4. **레이트 제한 준수**: machine 동시 실행 제한을 지킨다
5. **기밀 데이터 주의**: machine에 API 키·비밀번호를 넘기지 않는다

## Phase 1: P&L 검증(machine 활용)

### 용도와 instruction 템플릿

#### 1-A: P&L 독립 계산

```
다음 트레이드 로그에서 P&L을 독립 계산하라:

로그 파일: {경로}
기간: {시작일〜종료일}
계산 항목:
- 총 P&L(수수료 포함)
- 승률(승/패/무승부 건수)
- 프로핏 팩터
- 최대 드로다운(액·율·기간)
- 최대 연속 패배 수
- Sharpe 비율

비교 대상: {Director/Engineer가 보고한 수치}
출력: 계산 결과 + 보고값과의 차이 목록
```

#### 1-B: 기대값 vs 실적 괴리 분석

```
다음 백테스트 결과와 라이브 실적을 비교하라:

백테스트: {backtest-report.md 경로}
라이브 실적: {트레이드 로그 경로}
비교 항목:
- P&L 괴리율
- 승률 괴리
- 최대 드로다운 괴리
- 거래 빈도 괴리
- 슬리피지 실적 vs 가정

출력: 괴리 요약 + 괴리가 임계 `{pl_divergence_threshold}`를 초과하는 항목 플래그
```

## Phase 2: 운용 건전성 스캔(machine 활용)

#### 2-A: 프로세스·연결 확인

```
다음 서비스의 가동 상태를 확인하라:

확인 대상:
- {bot명1}: 프로세스 가동 확인 + 최종 로그 타임스탬프
- {bot명2}: 프로세스 가동 확인 + 최종 로그 타임스탬프
- API 연결: {거래소명} API 응답 확인

출력: 각 서비스 상태(가동 중/정지/무응답) + 최종 활동 시각
```

#### 2-B: 자산 잔고 대조

```
다음 자산 잔고를 대조하라:

대조 대상:
- {거래소/지갑1}: 기대 잔고 {금액} vs 실 잔고
- {거래소/지갑2}: 기대 잔고 {금액} vs 실 잔고
- 합산 포트폴리오: 기대 합계 {금액} vs 실 합계

허용 차이: `{balance_tolerance}`

출력: 대조 결과 + 허용 차이를 초과하는 항목 플래그
```

#### 2-C: 체결 검증

```
다음 기간의 주문과 체결을 대조하라:

기간: 최근{기간}
대상: {bot명/전략명}

검증 항목:
- 발주된 주문 수 vs 체결한 주문 수
- 부분 체결 유무와 영향
- 체결 거부/타임아웃 유무
- 가정 슬리피지 vs 실제 슬리피지

출력: 체결 요약 + 이상 플래그
```

## Phase 3: Carry-forward 검증(machine 활용)

#### 3-A: Tracker 정합성 체크

```
다음 추적표의 정합성을 검증하라:

performance-tracker: {경로}
ops-issue-tracker: {경로}

검증 항목:
- 전회 리뷰에서 「미해소」였던 항목이 전건 언급되었는가(silent drop 검출)
- 상태 전이가 논리적인가(예: 「해소」→「미대응」으로의 역행이 없는가)
- 신규 추가 항목에 최초 검출일·심각도가 붙어 있는가

출력: 정합성 체크 결과 + silent drop 검출 목록
```
