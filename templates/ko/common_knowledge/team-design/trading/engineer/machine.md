# Trading Engineer — machine 활용 가이드

Trading Engineer가 machine으로 bot 구현·백테스트·인프라 구축을 수행하기 위한 가이드.

> **중요**: 계획서 없는 machine 실행은 금지. 반드시 MD 형식 계획서를 사전에 작성해 instruction으로 넘긴다.

---

## 기본 규칙

1. **계획서 우선**: machine에 넘기기 전에 계획서(impl.plan.md 상당)를 작성한다
2. **출력은 draft**: machine 출력을 그대로 본운용에 하지 않는다. Engineer가 검증한다
3. **파일 경로**: `state/plans/`에 계획서를 저장하고 machine에 넘기는 instruction으로 쓴다
4. **워크스페이스 지정**: machine에는 반드시 워크스페이스 디렉터리를 지정한다(`-d {path}`)
5. **레이트 제한 준수**: machine 동시 실행 제한을 지킨다
6. **본환경 직접 변경 금지**: machine에 본환경을 직접 변경시키지 않는다

---

## Phase 1: 구현 계획(Engineer가 작성)

strategy-plan.md를 받아 기술적 구현 상세를 계획서로 정리한다.

### impl.plan.md 템플릿

```markdown
# 구현 계획서: {전략명/태스크명}

status: draft
author: {anima명}
date: {YYYY-MM-DD}
type: impl-plan
source: {strategy-plan.md 경로}

## 개요
{strategy-plan.md 목표에 대한 기술적 접근 — 1〜3문}

## 구현 스텝
1. {스텝1: 구체적 구현 내용}
2. {스텝2: 구체적 구현 내용}
3. ...

## 테스트 계획
- [ ] {테스트1: 유닛 테스트 대상}
- [ ] {테스트2: 통합 테스트 대상}
- [ ] {테스트3: dry-run 검증}

## 리스크 파라미터 구현
| 파라미터 | strategy-plan.md 값 | 구현 방법 |
|-----------|---------------------|---------|
| 최대 드로다운 | {값} | {어디에 구현할지} |
| 포지션 상한 | {값} | {어디에 구현할지} |
| 손절 규칙 | {조건} | {어디에 구현할지} |

## 롤백 계획
{문제 발생 시 복구 절차}
```

## Phase 2: bot 구현(machine 활용)

### 용도와 instruction 템플릿

#### 2-A: 전략 로직 구현

```
다음 구현 계획에 따라 전략 로직을 구현하라:

계획서: {impl.plan.md 경로}
워크스페이스: {리포지토리 경로}

구현 요구:
- {언어/프레임워크}
- 랜덤 시드 고정(재현 가능성)
- 로그 출력(진입/청산/손익 기록)
- 에러 처리(API 장애 시 재시도, 포지션 보호)

테스트:
- 유닛 테스트를 작성할 것
- 기존 테스트가 전건 통과할 것
```

#### 2-B: 백테스트 실행

```
다음 전략의 백테스트를 실행하라:

전략: {코드 경로}
데이터: {데이터 경로/취득 방법}
기간: {시작일〜종료일}
파라미터: {파라미터 세트}

백테스트 조건:
- 슬리피지: {가정값}
- 수수료: {거래소 수수료율}
- 초기 자금: {금액}

출력:
- P&L 추이
- 승률·프로핏 팩터
- 최대 드로다운(액·율·기간)
- 거래 횟수와 평균 보유 기간
- Sharpe 비율
```

## Phase 3: 인프라 변경(machine 활용)

### 용도: 배포·설정 변경

```
다음 인프라 변경을 수행하라:

계획서: {impl.plan.md 경로}
대상: {서버/서비스}

변경 내용:
- {변경1}
- {변경2}

검증 절차:
- {검증1: 변경 후 동작 확인 방법}

롤백 절차:
- {문제 발생 시 복구 방법}
```

> **주의**: 본환경 변경은 dry-run으로 검증된 뒤여야 한다. machine에 본환경 API 키를 넘기지 않는다.
