# Writer Specialty Guidelines

## Document Structure Principles

### Reader-Oriented Structure
- Put the most important information first (inverted pyramid)
- Structure around what the reader wants to know
- Add table of contents and overview section for long documents
- Limit hierarchy to 3 levels (deeper nesting hurts readability)

### Document Skeleton
1. **Title**: Summarize content in a phrase
2. **Summary**: State document purpose and conclusion in 3 sentences or less
3. **Body**: Expand details in logical order
4. **Conclusion**: Recap key points and next actions

## Communication Guidelines

### Internal Documents
- Prioritize brevity. Skip redundant preamble
- Use team-standard terminology when there is shared understanding
- Use bullet lists so key points are visible at a glance
- Include owner and deadline for action items

### External Documents
- Add explanations for technical terms on first use
- Keep tone polite but not verbose
- Maintain consistent organizational tone
- Verify facts and check for typos before release

### Messaging
- Make subject/title convey the matter
- Minimize boilerplate greetings
- State requests clearly with deadlines

## Self-Review Checklist

Before finalizing, verify:

### Content
- [ ] Purpose is clearly stated at the start
- [ ] No logical gaps
- [ ] Explanation level matches reader's background
- [ ] Nothing essential is missing
- [ ] No unnecessary information

### Expression
- [ ] Sentences not too long (target: under ~60 chars per sentence)
- [ ] No unnecessary repetition
- [ ] Not overly passive voice (prefer active)
- [ ] Vague wording ("etc.", "various") made concrete where possible

### Format
- [ ] Heading hierarchy is consistent
- [ ] Bullet granularity is uniform
- [ ] Code blocks, links, tables render correctly
- [ ] No typos or grammar errors

## Clarity and Brevity

### One Idea Per Sentence
- Put one piece of information in each sentence
- Avoid over-connecting sentences with conjunctions

### Concreteness
- Prefer "reduced response time by 40%" over "significantly improved"
- Prefer "1,200 monthly active users" over "many users"
- Prefer numbers and proper nouns over vague expressions

### Avoid Redundancy
- "it is possible to ~" → "can ~"
- "perform ~" → "~"
- "~ is better" → "~" (when not comparing)
- Use "basically", "in principle" only when truly needed
