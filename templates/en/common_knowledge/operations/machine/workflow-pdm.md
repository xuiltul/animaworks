# Machine Workflow — PdM (Product Manager)

## Role Definition

PdM uses **machine as a tool for investigation, while making planning decisions themselves**.

- Investigation & information gathering -> delegate to machine
- Interpreting results, prioritization, deciding implementation approach -> PdM's own judgment
- Creating the plan document (plan.md) -> PdM writes it themselves

> Prerequisite: Understand the meta-pattern and common principles in `operations/machine/tool-usage.md`.

## Phase 1: Investigation

### Step 1: Create an investigation plan (PdM writes this)

Create a plan document that clearly defines what machine should investigate.

```bash
write_memory_file(path="state/plans/{date}_{summary}.investigation.md", content="...")
```

### Step 2: Delegate investigation to machine

```bash
animaworks-tool machine run \
  "$(cat $ANIMAWORKS_ANIMA_DIR/state/plans/{investigation_plan})" \
  -d /path/to/worktree
```

Machine returns investigation results as text.
Save results as `state/plans/{date}_{summary}.investigation.md` (`status: draft`).

### Step 3: Verify investigation results

PdM reads investigation.md and confirms:

- [ ] Does it contain answers to the investigation objectives?
- [ ] Are facts and speculation clearly distinguished?
- [ ] Are there any important information gaps?
- [ ] Is additional investigation needed for any areas?

Fix or supplement as needed, then change to `status: approved`.

## Phase 2: Plan Creation

### Step 4: Create plan.md (PdM's own judgment)

Based on investigation.md, **PdM themselves** creates plan.md.

The "Implementation Approach", "Priorities", and "Constraints" in plan.md are the core
of PdM's judgment and MUST NOT be written by machine (NEVER).

### Step 5: Delegate the plan

Confirm `status: approved` on plan.md and pass to Engineer via `delegate_task`.

## Investigation Plan Template

```markdown
# Investigation Plan: {Problem/Theme Summary}

status: draft
author: {anima_name}
date: {YYYY-MM-DD}
type: investigation

## Investigation Objective

{What needs to be clarified — 1-3 sentences}

## Investigation Targets

{Target code, log files, external services, time ranges, etc.}

- {target1}
- {target2}

## Investigation Steps

{Specific steps for machine to execute}

1. {step1}
2. {step2}
3. {step3}

## Judgment Criteria

{What findings indicate "problem exists" or "action required"}

## Expected Output

{Information and format to include in the investigation report}

- List of findings (clearly distinguish facts vs. speculation)
- Impact assessment
- Recommended actions
```

## Plan Template (plan.md)

```markdown
# Plan: {Task Name}

status: draft
author: {anima_name}
date: {YYYY-MM-DD}
type: plan

## Goal

{Objective to achieve — 1-3 clear sentences}

## Background & Investigation Results

{Summary of investigation.md. Evidence for why this work is needed}

## Target Files

{List of files to change or affected}

- {file1}
- {file2}

## Current State Analysis

{Understanding of relevant code. Problem causes and code structure}

## Implementation Approach

{Specific change plan in steps — PdM's own judgment}

1. {approach1}
2. {approach2}
3. {approach3}

## Constraints

{Coding standards, API compatibility, performance requirements, etc.}

- {constraint1}
- {constraint2}

## Completion Criteria

{Objectively verifiable criteria}

- {criterion1}
- {criterion2}

## Test Strategy

{High-level test target and test case approach}

## Risks

| Risk | Mitigation |
|------|------------|
| {risk1} | {mitigation1} |
| {risk2} | {mitigation2} |
```

## Constraints

- Investigation plan MUST be written by PdM themselves
- Judgment sections of plan.md (implementation approach, priorities, constraints) MUST be written by PdM
- investigation.md generated by machine remains a draft until PdM verifies and approves
- NEVER pass a plan.md without `status: approved` to Engineer
