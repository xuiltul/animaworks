# ライター専門ガイドライン

## ドキュメント構造の原則

### 読者を意識した構成
- 最も重要な情報を最初に置く（逆ピラミッド構造）
- 読者が「何を知りたいか」を起点に構成する
- 長いドキュメントには目次と概要セクションを設ける
- 階層は3レベルまでに抑える（深すぎる階層は読みにくい）

### 文書の骨格
1. **タイトル**: 内容を一言で表す
2. **概要**: 3文以内で文書の目的と結論を述べる
3. **本文**: 論理的な順序で詳細を展開
4. **まとめ**: 要点の再確認とネクストアクション

## コミュニケーションガイドライン

### 社内文書
- 簡潔さを優先する。冗長な前置きは不要
- 専門用語はチーム内で共通認識があれば使ってよい
- 箇条書きを活用し、一目で要点が分かるようにする
- アクションアイテムには担当者と期限を明記する

### 対外文書
- 専門用語には初出時に説明を添える
- 丁寧だが冗長にならないバランスを保つ
- 会社・組織としての一貫したトーンを維持する
- 公開前に事実確認と誤字脱字チェックを徹底する

### メッセージング
- 件名/タイトルだけで用件が伝わるようにする
- 「お疲れ様です」などの定型挨拶は最小限にする
- 依頼事項は明確に、期限付きで記載する

## セルフレビューチェックリスト

文書を完成させる前に、以下を確認する:

### 内容
- [ ] 目的が冒頭で明確に述べられているか
- [ ] 論理の飛躍はないか
- [ ] 読者の前提知識に合った説明レベルか
- [ ] 必要な情報が漏れていないか
- [ ] 不要な情報が含まれていないか

### 表現
- [ ] 一文が長すぎないか（目安: 60文字以内）
- [ ] 同じ内容の繰り返しはないか
- [ ] 受動態が多すぎないか（能動態を優先）
- [ ] 曖昧な表現（「など」「いろいろ」）を具体化できないか

### フォーマット
- [ ] 見出しの階層が整理されているか
- [ ] 箇条書きの粒度が揃っているか
- [ ] コードブロック、リンク、表が正しく表示されるか
- [ ] 誤字脱字はないか

## 明瞭さと簡潔さの原則

### 一文一義
- 1つの文に1つの情報だけを入れる
- 接続詞で文を繋げすぎない

### 具体性
- 「大幅に改善した」ではなく「レスポンス時間を40%短縮した」
- 「多くのユーザー」ではなく「月間アクティブユーザー1,200人」
- 数値や固有名詞で裏付けられる表現を優先する

### 冗長表現の排除
- 「〜することが可能です」→「〜できます」
- 「〜を行う」→「〜する」
- 「〜の方が」→「〜が」（比較でない場合）
- 「基本的に」「原則として」→ 本当に必要な場合のみ使う
